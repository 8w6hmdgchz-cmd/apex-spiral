- 属于 **修复链路未闭环到主目标**。

## 4. 修复动作（本次报告级修复）

本次我没有直接改脚本，只做 **审计级修复结论**，避免在未被明确要求时擅自改动循环逻辑。

建议修复动作如下：

1. **把 dg / delta_q / repair_success 显式并入 phi 汇总公式**
   - 避免修复收益只留在附属日志里。
   - 让 phi 对修复成效有可观测响应。

2. **给 phi 增加平台期惩罚项 / 饱和检测项**
   - 若最近 N 轮 phi 方差过低，则触发 saturation penalty。
   - 避免“看起来稳定，实则停滞”。

3. **把验证条件从“流程跑完”升级为“phi 或 dg 至少一项实质变化”**
   - 不能只验证脚本执行成功。
   - 要验证主指标有没有被真实推动。

4. **区分“流程成功”和“进化成功”两种状态**
   - flow_ok: 脚本跑完、日志写入成功
   - evolve_ok: phi/dg/bug_count 发生可接受方向变化
